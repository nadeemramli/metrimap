## CRITICAL GOTCHAS (read before touching these areas)

1. **Prisma regen workflow.** `npm run prisma:types` is effectively **broken** for this repo: `schema.prisma` is **hand-curated** (relation fields pruned to avoid a `projects.tags` scalar-vs-relation name clash), and `prisma db pull` regenerates them, reintroducing a P1012 collision. **Correct workflow:** hand-edit `src/lib/prisma/schema.prisma` to add the new column, then run **`npm run prisma:generate`** ONLY (not `db pull`). (DATABASE_URL is now valid — the earlier P1000 was a stale password, since fixed.)
2. **`src/shared/lib/supabase/types.ts` was hand-refreshed, not fully regenerated.** I surgically added `projects.is_public/is_starred/archived_at` + the `canvas_nodes`/`source_connections`/`source_connection_secrets` blocks. The file still has **latent drift** elsewhere (e.g. a phantom `relationships.description` column, non-null `source_id`). A full `supabase gen types` overwrite is worth doing once, but will surface cascade type-fixes.
3. **New `projects` columns + the store.** The store's generic `updateProject` only maps a fixed camelCase field set and routes through prisma-zod. For new flags use dedicated direct-update services (`setProjectStarred`/`setProjectArchived` pattern in `services/projects.ts`).
4. **Warehouse `verify_jwt`.** `warehouse-proxy` is deployed with `verify_jwt: true`; the Clerk-JWT-through-gateway path is runtime-unverified. If warehouse calls 401 at the gateway, see [`data-source-caveats.md`](./data-source-caveats.md) Caveat 1 (flip to false + add in-function auth, or configure gateway for Clerk JWKS).
